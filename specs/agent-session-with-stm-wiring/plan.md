# Implementation Plan: STM Domain Model Refactor and Service-Layer Restructuring

**Branch**: `agent-session-with-stm-wiring` | **Date**: 2026-03-17 | **Spec**: [spec.md](./spec.md)
**Input**: Feature specification from `specs/agent-session-with-stm-wiring/spec.md`

## Summary

Refactor the STM identity model so that `conversation_id` (not `session_id`) becomes the canonical agent memory key, enabling multiple independent conversations per session. Implementation covers schema normalization (Phase A) followed by service-layer restructuring (Phase B), executed as a deliberate API break with clean-slate data migration.

## Technical Context

**Language/Version**: Python 3.8+
**Primary Dependencies**: Flask, LangGraph (`MongoDBSaver`), LangChain >=1.0.0, PyMongo
**Storage**: MongoDB 5.0 (collections: `workspaces`, `sessions`, `conversations`, `agent_checkpoints`), Redis 6.2 (caching)
**Testing**: pytest (unit + integration), protocol-based mocking
**Target Platform**: Docker Compose (local), AKS (production)
**Project Type**: Web service (Flask API + Socket.IO + background agent)
**Constraints**: No new REST CRUD endpoints (Phase C scope); no frontend changes; no LTM/vector store work

## Technical Decisions

The following decisions were made based on **Section 10: Risks and Technical Considerations** of the [AGENTIC_APP_WITH_STM_INTEGRATION_ROADMAP.md](../../docs/High-level%20Design/AGENTIC_APP_WITH_STM_INTEGRATION_ROADMAP.md):

| # | Topic | Decision | Rationale |
|---|-------|----------|-----------|
| В§10.1 | Backward Compatibility | **Deliberate API break** вҖ” Remove `session_id` from agent/chat entry points. Stateful callers must provide `conversation_id`; if omitted, FR-017 stateless mode is used (no persistence). No transitional shim. | Eliminates dual-path complexity while preserving stateless operation. Project is pre-production; no external consumers depend on the legacy API. User Story 5 (backward compat shim) is dropped from this implementation. |
| В§10.2 | Data Migration | **Clean-slate** вҖ” Wipe existing `conversations` and `agent_checkpoints` collections during migration. | Project stage is dev/test only. No production data exists. A promote-in-place migration adds complexity for zero-value data. |
| В§10.3 | Authorization & Isolation | **Normalize types first** вҖ” All cross-collection references (`workspace_id`, `session_id`, `user_id`) become `string` type in Phase A before any service work begins. | Prevents fragile type-coercion logic in service layer. Aligns with FR-003. |
| В§10.4 | Implementation Ordering | **Schema first, then services** вҖ” Complete all schema and repository changes before touching service layer code. | Clean boundary between data contract and business logic. Easier to test schema changes in isolation. |

### Impact on Spec

- **FR-019 (Backward Compatibility)** is **deferred** вҖ” not implemented in this iteration. Callers that send legacy `session_id` without `conversation_id` are not shim-resolved; stateless requests without either identifier remain supported per FR-017.
- **User Story 5** is **deferred** вҖ” the session-to-default-conversation resolution shim is not built.
- **FR-006 (Migration Path)** is simplified to a clean-slate drop+recreate strategy instead of promote-in-place.

| Field | Current | Target | Change |

#### 4. Index Changes

**conversations collection:**
- **Drop**: `idx_conversations_session_id_unique` (session_id unique) вҖ” no longer 1:1
- **Add**: `idx_conversations_conversation_id` (conversation_id, unique)
- **Add**: `idx_conversations_thread_id` (thread_id, unique)
- **Add**: `idx_conversations_session_id` (session_id, non-unique)
- **Add**: `idx_conversations_hierarchy_status` (workspace_id + session_id + status, compound)
- **Keep**: `idx_conversations_user_status`, `idx_conversations_workspace_status`, `idx_conversations_status_activity`

**sessions collection:**
- **Add**: `idx_sessions_session_id` (session_id, unique)
- **Keep**: `idx_sessions_workspace`, `idx_sessions_workspace_status`

```python
    """New protocol for session context access."""
    def get_session(self, session_id: str) -> Optional[Dict]: ...
    def get_session_context(self, session_id: str) -> Optional[Dict]: ...

class ConversationProvider(Protocol):
    """Updated: conversation_id instead of session_id."""
    def get_conversation(self, conversation_id: str) -> Optional[Dict]: ...
```

## Key Design Decisions

### Why conversation_id == thread_id (not a derived value)

Per [AGENT_MEMORY_TECHNICAL_DESIGN.md В§Decision 3](../../docs/domains/agent/AGENT_MEMORY_TECHNICAL_DESIGN.md):
- 1:1 mapping simplifies debugging and tracing
- No translation layer between application and LangGraph
- `conversation_id` is generated via `uuid.uuid4()` which is safe for MongoDB index and LangGraph thread_id

### Why session context lives on session, not conversation

Per [AGENT_MEMORY_TECHNICAL_DESIGN.md В§Decision 1](../../docs/domains/agent/AGENT_MEMORY_TECHNICAL_DESIGN.md):
- Session stores reusable assumptions, intent, and symbol focus across conversations
- Conversations inherit session context but can apply thread-local overrides
- This prevents duplication and ensures siblings share the same analytical starting point

### Why clean-slate instead of migration

- No production data exists (dev/test stage)
- Avoids promote-in-place complexity for zero-value legacy records
- LangGraph checkpoint data does not need rekeying вҖ” it is simply dropped and recreated naturally

### Why deliberate API break instead of shim

- No external consumers of the current `session_id` API
- Eliminates dual-path resolution logic in ChatService
- Forces immediate alignment of all callers to the canonical identity model
- Reduces testing surface (no shim edge cases)

## Risk Mitigations

| Risk | Mitigation |
|------|------------|
| Existing tests break after schema changes | Phase 1 (schema) runs first; tests updated in Phase 3 before CI enforcement |
| SessionService adds circular deps | Use protocol-based DI (`SessionProvider`); no direct import of `SessionService` in ChatService |
| Agent checkpoint collections have write-ahead logs | Clean-slate migration drops all LangGraph-managed collections (checkpoints + writes) |
| Session context inheritance creates coupling | Context is a plain dict merged at ChatService level; SessionService and ConversationService remain independent |
| MongoDB schema validation rejects existing records after schema change | Clean-slate drop eliminates stale records; validation set to `moderate`/`warn` for safety |

## Complexity Tracking

No constitution violations to justify. All changes align with existing patterns (BaseService, MongoGenericRepository, Protocol-based DI, Factory injection).
